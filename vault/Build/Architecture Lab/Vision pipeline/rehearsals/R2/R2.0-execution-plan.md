## Procedure — level A (unsaved), evidence-first

Run the frontend from **this** worktree (post-rebase) so the probe reflects the rehearsal branch.

1. Capture the **pre-state** read-only: the post's `text_blocks` (expect `[]`), `percepts`,
   `region_annotations` — a hash/snapshot for later comparison.
2. Open `/posts/695be6c9a9ea58f1b6aef5e0`, enter the Writer/edit mode.
3. `/percept` slash command → insert the **`pctx_mrpi3rjk_0`** chip into local editor state.
4. Click the chip → observe recall play on the image.
5. **Discard without saving** (level A: no persistence).
6. Re-read the post and confirm `text_blocks` is still `[]` and percepts/regions are byte-identical.

Fixtures are only permitted if the real post cannot render; if used they must be labelled
fixture-rendered, never live.

## Pass / fail criteria

**PASS** — all of: the slash menu offers the percept; the chip inserts and renders with its
`data-*` attributes; clicking it triggers recall visibly on the image; no console errors; and the
production post is **unchanged** after discard.

**FAIL (report, do not repair in R2.0)** — any of: the `/percept` item is absent or lists no
percept; insertion or chip render fails; the click does not drive recall; recall throws; or the
post mutated despite not saving. A broken loop is a legitimate, valuable R2.0 result — R1's own
doctrine: *a stopped/degraded run is reported, never fabricated as success*.

**BLOCKED** — if servers or the browser harness fail again, report the exact blocker; do not
substitute unit tests for rendered proof (the logic is already covered by 46 existing frontend
tests — what is missing is precisely the rendered evidence).
