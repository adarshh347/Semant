# P2.2 — Surprises & Corrections

## Surprises

1. **The three Differential hooks discard `run_id`.** P2.1 added `run_id` to the route
   responses, but `useMaskRefine`/`useSemanticRead`/`useFindSimilar` return `{region}`/
   `{semantics}`/`{results}` and never read it. Rather than edit the hooks (their result/error
   semantics are frozen by the task), the rail derives its **immediate-refresh signal** from the
   hooks' existing `status` values (already available in `DifferentialWorkspace`), refreshing
   when any transitions to a terminal status. Zero hook edits — cleaner than exposing `run_id`.

2. **All four `latest` projections already embed `events`.** So the `{run_id}` endpoint is
   unnecessary; four bounded `latest` reads give the full picture. The rail never calls
   `{run_id}`.

3. **The frontend test harness has no DOM** (node vitest, no jsdom/`@testing-library`, no
   component tests anywhere). This *forced a good shape*: all honest-mapping/copy logic is pure
   and unit-tested; the JSX is thin and rendered-verified in a browser. No new test
   infrastructure was introduced.

4. **`vite dev` is blocked by `ENOSPC`** (inotify watch limit — five worktrees × node_modules),
   not the older exit-144 issue. Corrected by using `vite build` (no watch) of a small fixture
   harness + a short-lived static server, which renders the real component in a real browser.

5. **No live vision-run data exists.** The DB currently has **0 vision_runs** (P2.0/P2.1 cleaned
   up their disposable data), and there is no `.env`/backend in this worktree. Generating live
   runs would mutate real posts (explicitly forbidden). So every rendered state is honestly
   labeled **fixture-rendered**, not live.

## Corrections / deliberate scope holds

- **`refine` unavailability is not directly observable** within the allowed data boundary: a
  SAM2 503 mints **no run**, so `latest?operation=refine` returns `{run:null}` — indistinguishable
  from “refine never run”. The rail therefore renders refine-null as a quiet “No recorded
  activity” (not a false “unavailable”), and renders an explicit `unavailable` **only** when a
  run actually carries `status:"unavailable"`. Honest limitation, documented; not worked around
  by querying a capabilities endpoint (out of the §3 data boundary).
- **Cross-operation shared-region co-reference is deliberately not surfaced as linkage.** Each
  operation is a standalone block; per-entry “Referenced <part>” is shown, but the rail draws no
  line/arrow between two runs that reference the same region — that would edge toward implying
  causality.
- **Harness scaffolding was removed** from `frontend/` after capture (kept, reproducibly, under
  `evidence/railharness/`); it is not part of the production diff.

## Does the evidence suggest a genuine future need for cross-run correlation?

**Modestly, and only the non-causal kind.** Building and rendering the rail made two things
clear:
- The four-operation snapshot is **legible on its own**. A curator can already see, per run,
  what ran, its status, its stages, and which region it referenced — without any cross-run link.
- The one place a curator would plausibly want *more* is **region-scoped observational
  grouping** — “show me every recorded operation that referenced this part” — which today
  requires eyeballing the per-entry region refs. That is exactly the **non-causal** enhancement
  the P2.1 correlation investigation already recommended (optional first-class *indexed*
  `region_id`, and recording the *read* `geometry_rev` on semantic/retrieval runs).
- There is **no** evidence of a need for **causal** correlation (a Passage entity /
  `parent_run_id` / cross-run `dependencies`). Nothing in the rail wants to assert “B happened
  because of A”, and the honest posture (co-reference, not causation) held comfortably.

Recommendation (not built): if a future increment pursues correlation, scope it to
region-anchored **observational grouping**, reviewed separately — not a causal/passage schema.
