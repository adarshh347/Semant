## Rendered verification (browser) — **rendered-verified**

Long-lived `vite dev` is blocked here by an **`ENOSPC` inotify watch limit** (many worktrees),
and there is **no running backend** in this worktree (no `.env`; DB currently has 0 vision_runs).
Per §9's fallback, the existing short-lived/local browser harness was used: a `vite build`
(no watch) of `railharness/` rendering the **real** `VisionActivityRail` (real CSS + `index.css`
tokens) inside `.diff-inspector`-width columns (280 px and 208 px), with a stubbed `window.fetch`
returning fixture run projections. Served via a short-lived static server; driven with the
browser tools.

**All states are fixture-rendered** (labeled honestly). No state is live-rendered this session —
no backend/runs available and generating runs would mutate posts (forbidden here).

## Layout safety (image tools / semantic reading / similarity / region highlighting)

The rail is a child of the **inspector grid column** (`grid-template-areas:"rail stage insp"`),
structurally isolated from the image `stage` column — it **cannot** obscure the image, tool
rail, or overlays by construction. The harness confirms it sits quietly at the bottom of an
inspector-width column beneath a representative tool panel without pushing or overlapping it, at
both widths. Selected-region highlighting is driven through the untouched `store`/`RegionOverlay`
in the stage column; the rail neither reads nor writes selection or curator state.

## Database / post mutation check

The rendered verification used a **fully stubbed `window.fetch`** in the browser; **no backend
process ran** in this worktree (confirmed: no `.env`, config load fails). The rail issues only
`GET` requests to read-only endpoints and performs no writes. Therefore no post, curator-state,
or DB mutation occurred or is possible from the rail (nor from merely opening it).
