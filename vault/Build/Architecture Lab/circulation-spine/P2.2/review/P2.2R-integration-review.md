## Verdict

**GO-WITH-BLOCKING — not merge-ready.** The rail is well-built, honest in copy, and correctly
degrades on missing fields — but it **conflates "could not read activity" with "no activity"**
(B1), which directly violates the increment's core honesty contract (§3). B1 is a small,
well-scoped fix. Everything else is Recommended/Optional. Do **not** claim the *integrated*
Differential workspace rendered-verified: only the component-in-harness was rendered.

## Blocking

### B1 — Absence / fetch-failure conflation (honesty defect) · blocks merge
**Files:** `visionActivity.js:42-58` (esp. `:48`, `:51-53`); `useVisionActivity.js:29-37`,
`:75`; `VisionActivityRail.jsx:91`, `:28`, `:120`.
**Evidence:** `fetchAllRuns` maps **every** non-abort failure to `[op, null]` — `!res.ok`
(401/403/404/500) at `:48`, `res.json()` malformed at `:49→:51`, and network rejection at
`:51-53`. Only `AbortError` re-throws. So `Promise.all` never rejects except on abort, and the
hook's `catch`/`setError('unavailable')` (`useVisionActivity.js:34`) is effectively **dead code**.
Worse, the component destructures `{ runs, loading, refresh }` (`VisionActivityRail.jsx:91`) and
**never reads `error`** (returned at `useVisionActivity.js:75`). Net: a failed read is
indistinguishable from a real `{run:null}` and renders as `No recorded activity`
(`:28`) / `nothing recorded yet` (`:120`).
**Realistic scenario:** the posts router requires auth — `app.include_router(posts.router, …,
dependencies=[Depends(require_api_key)])` (`backend/main.py:91`) — and the frontend only attaches
`X-API-Key` **when `VITE_API_KEY` is set** (`frontend/src/config/api.js`). A deployed backend with
a key + a build without `VITE_API_KEY` ⇒ every vision-run read returns **401** ⇒ the rail silently
says "nothing recorded yet." Same for backend down (network error) or a 500.
**Impact:** the surface whose entire purpose is honest observation lies by omission exactly when
the truth is "I couldn't read this." User-visible, on the default (collapsed) header too.
**Smallest safe correction:** have `fetchAllRuns` return a per-operation discriminated result
(e.g. `{op, run, failed:true}` on a caught non-abort failure instead of `null`), thread a
per-op/overall `error` into the component, and render a distinct quiet "Couldn't read activity"
state (not alarming, not "No recorded activity"). Preserve partial success (a single failed op
must not blank the other three — currently true and must stay true). Add one unit test asserting
failure ≠ absence. **Blocks merge and live verification** (live verification must exercise this
state).

## Recommended

### R1 — `relTime` is timezone-naive; deriveEntry never exercised against a real HTTP response
`visionActivity.js:185-187` (`Date.parse(iso)`). The **only committed "real run JSON"**
(`P2.0/evidence/p2.0-retrieved-run.json`) stores `created_at:'2026-07-21 01:11:34.540000'` — a
Python `str(datetime)` (space, no `T`, no tz). `Date.parse` treats that as **local** time
(verified: → `2026-07-20T19:41:34Z`, off by the tz offset) rather than the ISO the fixtures use.
The live wire format is ISO (FastAPI `jsonable_encoder`), so production is likely fine — but there
is **no committed proof** the frontend consumes the real response shape; the fixtures are
hand-authored to ISO. **Correction:** capture one real `GET …/vision-runs/latest` response into a
fixture and run it through `deriveEntry`; make `relTime` tolerant of tz-naive/space timestamps (or
assert the wire format). Degrades safely today (bad date → age omitted), so not blocking.

### R2 — Unconditional fetch on mount, even while collapsed
`VisionActivityRail.jsx:91` calls `useVisionActivity` unconditionally; its mount effect
(`useVisionActivity.js:41-49`) fires **4 requests on every Differential open**, whether or not the
curator ever expands the rail. Bounded (once per post) but wasteful and — with B1 — means every
post-open silently swallows read failures. **Correction:** defer the first fetch to first expand
(lazy), keeping the collapsed header a pure 1-line control.

### R3 — Real Differential layout is structurally inferred, not rendered
Screenshots are the **fixture harness** at inspector widths, not the real workspace. Two concrete
CSS risks are unverified there: (a) `.va-rail { margin-top:auto }` (`VisionActivityRail.css:10`)
on a child mounted **after** `.diff-insp-counts` (`DifferentialWorkspace.jsx:814` then `:821`) can
push the rail to the column bottom leaving a gap under the footer, or below the fold; (b) the
inspector's container-query reflow to a `max-height:42vh` bottom row at narrow **height** (not
just width) was never rendered. §7's "does not obscure image tools" is *structurally* sound (the
grid isolates the `insp` column from `stage`), but layout-fit is inferred. **Correction:** one
real-workspace render (normal + short viewport) before merge.

### R4 — "monkey-patch applies" claim is conditional
`P2.2-implementation-report.md` (Data boundary) says reads go through `API_URL` "so the config's
X-API-Key monkey-patch applies." Per `config/api.js`, the patch is installed **only if
`VITE_API_KEY` is set**. State the condition; it compounds B1 in keyed deployments.

### R5 — Test-evidence classification overstates lifecycle coverage
The 19 tests (`visionActivity.test.js`) directly cover pure logic + `fetchAllRuns` orchestration +
abort *propagation*. But the **hook's** "post change aborts old requests" and "unmount clears
timers" (`useVisionActivity.js:41-49`, `:53-60`) are **code-inferred only** — no post-switch or
unmount was unit-tested or browser-observed (the harness renders static rails, never switches post
or unmounts). The evidence table maps case 13 to "fetchAllRuns abort (unit)", which is a
*different* abort than the hook's post-change abort. **Correction:** relabel 13/14 as
structurally-inferred, or add a jsdom/renderHook test.

## Contract fidelity

Faithful overall. `statusPresentation` (`visionActivity.js`) covers **all nine** `JobStatus`
values (pending/running/succeeded/partial/skipped/unavailable/timed_out/cancelled/failed) + the
`stale` flag + a safe default. `deriveEntry` reads only real projected fields — `status`, `stale`,
`events[]` (`stage_id/status/latency_ms/adapter/fallbacks/dependencies/error`), `actual_source`,
`completed_at/updated_at/created_at`, `terminal_reason`, `error`, `telemetry_degraded`,
`result_summary.{neighbours,space}`, `requested_profile.{region_id,region_ids}` — all matching the
committed backend projection and the real run JSON (`P2.0/P2.1` evidence) event/run keys. Missing/
legacy/unknown fields degrade safely (empty events, absent age, unknown status → muted). Honesty
holds: retrieval is "Research neighbours — no relation created" (never truth), semantic is
"Interpretation only — geometry unchanged" (never geometry), and separate runs render as
standalone blocks with no cross-run link (rendered-scan confirmed no causal wording). **Gaps:**
date-format robustness (R1); fixtures are hand-authored to the ISO wire shape rather than the
committed Python-dumped shape, so wire-shape flow is inferred, not proven.
