## R1 — cancellation now terminalizes the run `[implemented]`

**Routes (all four instrumented operations: `dissect`, `refine`, `semantic_read`,
`find_similar`).** Each now has an explicit `except asyncio.CancelledError` **before**
`except Exception`, which finalizes the already-minted run as
`JobStatus.CANCELLED` with `terminal_reason="request_cancelled"` and then **re-raises the
original cancellation**. No generic `BaseException` catch; cancellation is never swallowed or
translated.

**Protected boundary covers every await after run creation.** The initial event write(s)
(`*.receive`, and refine's `*.propose`) were moved *inside* the `try`, so a cancellation
delivered during the very first post-mint await is caught and terminalized rather than
escaping. (400/404/503/422 guards still run *before* the run is minted, so rejections remain
run-free.)

**`VisionRunRecorder.finish` hardened.** The finalize is now wrapped in `asyncio.shield(...)`,
and `_finished` is set only *after* a finalize attempt (or on the `CancelledError` path):
- The shield keeps the **short telemetry-finalization write** running to completion even when
  the surrounding request is cancelled mid-write — so a cancellation during finish's own DB
  await can no longer abort the write and strand the run. Only this telemetry op is shielded;
  **model and route work are never shielded**.
- On `CancelledError`, `finish` marks `_finished` (so a retry can't double-finalize) and
  re-raises — cancellation still propagates.
- The write-behind guarantee is preserved for ordinary failures: non-cancellation exceptions
  are swallowed and flagged `telemetry_degraded`; `finish` never raises them into the route.

**Documented residual (unchanged, honest).** If the shielded finalize write itself fails
(database down) or the event loop is torn down before it commits, the run can still be left as
a **stale `RUNNING`** record — surfaced honestly by the read projection's staleness flag, never
silently "completed". No reaper was added (explicitly out of scope).
