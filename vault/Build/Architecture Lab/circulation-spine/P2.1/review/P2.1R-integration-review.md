- **Query/index alignment.** The only non-`_id` query (`get_latest_run` by `post_id` +

- **Bounded-payload enforcement.** `requested_profile`/`detail`/refs for the siblings are

## RECOMMENDED

### R1 — Cancellation / shutdown leaves runs stuck `RUNNING` (all four routes)
Directly answering the review's focus: **yes, a run can remain unintentionally `RUNNING`.**
All four routes finalize via `except Exception` only — there is **no `finally` and no
`BaseException`/`asyncio.CancelledError` handling** anywhere (grep-confirmed). If a request is
cancelled (client disconnect, timeout, server shutdown) between `rec.start()` and the
completion of `rec.finish()`, `CancelledError` (a `BaseException`) propagates **past** the
`except Exception`, so `rec.finish` never runs and the run is left `RUNNING` forever.
- *Impact:* telemetry hygiene only — route behavior, geometry, curator state, and the primary
  cancellation are all unaffected; and the read side is honest (a `RUNNING` run older than
  `STALE_AFTER_SECONDS` projects `stale` with a real age). But the run never reaches a
  terminal status, violating Invariant 13, and stale `RUNNING` docs accumulate unboundedly
  over time.
- *Second, documented path to the same state:* if the finalize write itself fails, `finish`
  swallows it and the run stays `RUNNING` (P1's "unavoidable case", also stale-projected).
- *Escalation:* **BLOCKING** iff the acceptance bar requires every created run to reach a
  terminal status under cancellation/shutdown.
- *Suggested fix (not applied):* wrap the body in `try/…/finally` that finalizes-if-not-yet-
  terminal, or add `except BaseException:` that marks the run `CANCELLED` (a terminal
  `JobStatus`) and re-raises. The recorder already exposes idempotent finalize, so a
  finally-style `rec.finish(...)` guarded by its `_finished` flag is safe.

### R2 — Projected error text is bounded but not sanitized (mild info-leak)
Run/event `error` is truncated to 1000 chars (good) but is raw `str(e)`. An httpx fetch
failure can embed the `photo_url`; other exceptions can embed hostnames/paths. This is stored
in Mongo and returned by the **unauthenticated** `/vision-runs/{run_id}` read route (the whole
post API is authless — systemic, pre-existing). Anyone who can reach the API and knows a
`post_id`+`run_id` can read it.
- *Fix (not applied):* scrub/whitelist the *projected* error (keep the raw string in server
  logs), and/or gate the vision-run read routes behind whatever auth the app adopts. Aligns
  with the already-documented P2 rule "raw errors are never the default user-facing message".

### R3 — Missing tests for the siblings' except-path FAILED finalization
The P2.1 suite covers run creation, correlation refs, and telemetry-failure-at-*start*
(`run_id: null`), but not a mid-body exception → run finalized `FAILED` + primary re-raised
(the P1 suite has this only for `dissect`). Add one per sibling, mirroring
`test_route_reraises_primary_exception_and_marks_run_failed`. (Would also naturally host a
regression test for R1 once fixed.)
