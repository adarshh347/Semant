## Chosen smallest representative set (one per family)

- **Geometry → `refine-region/confirm`.** The one route that *authors* authoritative geometry
  (`mask_rle` + `geometry_rev`) and persists `region_annotations`. Highest-stakes for the
  "do not change authoritative geometry / curator state" invariant.
- **Semantic → `semantic-read`.** Pure semantic writer: touches `semantics` only, never
  geometry; `merge_curator_state` protects sticky decisions. (Preferred over `enrich-regions`,
  which mutates `region_annotations` labels + writes the embedding sidecar — a heavier,
  geometry-adjacent surface; kept out of P2.1.)
- **Retrieval → `find-similar`.** Read-mostly research; may index the region on demand
  (sidecar only), returns neighbours + distances, writes no post geometry/curator state.

These three are genuinely heterogeneous: a region-scoped geometry *writer* bearing
`geometry_rev`; a post-scoped semantic *writer* with no `geometry_rev`; a region-scoped
retrieval *reader* that writes only a sidecar. One instrumentation contract must fit all three
— which is exactly the P1.1 neutrality claim under test.

## Seam map (where events attach — clean `await` boundaries)

**refine/confirm** (via shared `_propose_refined_region`): validate+load → SAM2
`refine_session.preview` (mask compute) → merge-into-`region_annotations` (curator-preserving)
→ `update_one`. NB the 404/503/422 guards live *inside* `_propose_refined_region`, so minting
the run *after* that call returns keeps rejections run-free (matches P1's "no run for
rejections"); the propose latency is measured across the real call.

**semantic-read**: validate+load(404) → cached image fetch → `semantic_pass.run_semantic`
(VLM) → `merge_curator_state` → `update_one({semantics})`. Mint after post-load (P1 pattern).

**find-similar**: validate+load(404) → domain route + `retrieval_service.route` (space/model)
→ scope (`distinct post_id`) → cached image fetch → `find_similar_service.find_similar_for_region`
(index-on-demand + retrieve). Mint after post-load.

## Conflicts / hazards identified

1. **Shared helper `_propose_refined_region`** is used by BOTH preview and confirm. Instrument
   only confirm (the persisting one); do not touch the helper's control flow → preview stays
   byte-identical, and confirm's geometry/curator logic is unchanged.
2. **Availability 503s** (`FashionCLIP`/`SAM2` unavailable) and **422** (no prompt) are raised
   after post-load. Minting refine's run *post-propose* avoids creating runs for these; for
   semantic/find-similar there are no availability guards, so post-load minting is clean.
3. **`find-similar` response is a service-built dict** whose shape varies by status
   (`ready|empty|unavailable|error`). Adding `run_id` must be defensive (`setdefault` on a
   dict) and must not perturb `results`/`query`.
4. **Recorder naming.** The P1.1 write-behind helper is `DissectRunRecorder`. Reusing it
   verbatim for three non-Dissect operations is misleading; P2.1 generalizes it to
   `VisionRunRecorder` with a `DissectRunRecorder` alias (no contract change; P1 route + tests
   untouched).
5. **`operation` values must not collide** and must be distinct from `vision_service.*`
   adapters: `refine` / `semantic_read` / `find_similar` (never the internal service names).
6. **No new geometry/curator write** is introduced anywhere — instrumentation is additive
   telemetry + an optional `run_id` response field only.

## Correlation surface (full analysis in `P2.1-correlation-investigation.md`)

Available honest keys on the existing P1.1 contract: `post_id` (first-class), `operation`,
`created_at` (ordering), and — carried in the existing free-form `requested_profile` +
event `input_refs`/`output_refs` — `region_id`, `geometry_rev`, `base_geometry_rev`, `space`.
No new correlation schema, no cross-run causal `dependencies`. The irreducible gap (can't
*prove* run B causally followed run A without a forbidden correlation/parent-run id) is
reported, not papered over.
