## Corrections

### 1 — HTTP event projection aligned with the persisted contract `implemented`

`VisionStageEventOut` omitted `run_id` and `contract_version`, so FastAPI's response model
**stripped** them from `GET /vision-runs/{run_id}` even though persistence and the handoff
examples carry them. Both fields added. Now **one event contract** across persistence,
examples, and HTTP. *Test:* `test_response_model_retains_run_id_and_contract_version` builds
the real `VisionRunOut` from a projection and asserts both fields survive `model_dump()`.
(The handoff example JSONs already carried both fields — they are dumped from persistence —
so no example regeneration was required.)

### 2 — Cross-run event ownership rejected `implemented`

`append_event(run_id, event)` now: sets `event.run_id` when absent; **rejects** with
`EventRunMismatch` when present-and-different (an event is never persisted in run A while
claiming `run_id=B`); accepts when matching. The route-facing recorder still swallows the
error (write-behind intact). *Tests:* `test_append_event_sets_absent_run_id`,
`…_accepts_matching_run_id`, `…_rejects_conflicting_run_id` (asserts nothing persisted).

### 3 — Latest-run index aligned to the real query `implemented`

`get_latest_run` runs `find({post_id, operation}).sort(created_at, -1)`. `ensure_indexes`
now declares exactly one compound index **`(post_id, operation, created_at desc)`**
(`post_operation_created_idx`). The three earlier speculative indexes
(`post_id+created_at`, `status+updated_at`, bare `operation`) matched **no** real query and
were removed; single-run reads/appends/transitions key on `_id` (auto-indexed). A future P2
"list runs for a post" endpoint adds its own. *Test:*
`test_ensure_indexes_declares_only_the_supported_compound`.

### 4 — Persistence outcome recorded truthfully `implemented`

The route now captures the `region_annotations` `update_one` result. `matched_count > 0` →
`dissect.persist_regions = SUCCEEDED`; `matched_count == 0` → `= FAILED` (never a false
success) with `detail.matched_count` and an error string. The run then finalizes **PARTIAL**
with `terminal_reason = "persist_no_match"`. **Route response and control flow are
unchanged** — pre-P1 ignored this result and returned the computed regions, and so do we
(geometry, merge order, persistence payload, and primary-exception behavior all identical).
*Judgment call (documented):* a zero-match persist yields run **PARTIAL, not FAILED**,
because the HTTP route still returned a valid result; only the persist *stage* is FAILED.
*Test:* `test_zero_match_persistence_records_failed_not_success` (asserts response unchanged
+ stage FAILED + run PARTIAL/persist_no_match).

### 5 — Bounded telemetry payloads `implemented`

The recursive geometry gate became a bounded-shape gate, `assert_bounded_payload`, applied to
**`requested_profile`, `detail`, `provenance`, `input_refs`, `output_refs`, `fallbacks`,
`dependencies`, `result_summary`**. It rejects (as `BoundedPayloadError`): binary
(`bytes`/`bytearray`/`memoryview`); unsupported non-serializable types (custom objects, sets,
datetimes, ndarrays, whole Region objects — only `None`/`bool`/`int`/`float`/`str`/`list`/
`dict` pass); over-nesting; over-large collections/strings. Geometry keys still raise
`GeometryInEventError`. It is a plain recursive gate, **not** a serialization framework.

**Selected limits & why they suffice for an 8–13-event Dissect run:**

| Limit | Value | Rationale |
|---|---|---|
| `MAX_PAYLOAD_DEPTH` | 6 | Real `detail`/`provenance` nest ≤2; 6 is generous headroom. |
| `MAX_PAYLOAD_ITEMS` | 64 | Largest real payload is `Provenance` (~13 fields) / `detail` (≤6 keys). |
| `MAX_STRING_LEN` | 2048 | Source strings are ~40 chars; 2048 caps a pathological string. |
| `MAX_ERROR_LEN` | 1000 | See correction 6. |

Every value the route actually emits passes (`test_normal_route_payloads_stay_valid`);
boundary rejections covered by `test_bounded_payload_rejects_binary_and_unsupported`,
`…_rejects_excess_nesting_cardinality_and_length`, `…_applies_to_run_fields`. Masks, boxes,
image bytes, and complete Region objects therefore cannot enter Mongo or HTTP.

### 6 — Error transport bounded `implemented`

`bound_error()` truncates any stored/projected error (and `terminal_reason`) to
`MAX_ERROR_LEN = 1000` chars with a `…[truncated]` marker — applied in `make_event` (event
`error`) and `transition`/`finalize` (run `error`). **Truncated, never dropped:** the leading
technical detail (exception type + message head) is preserved for debugging; unbounded
provider output can never reach Mongo/HTTP. *Test:* `test_error_text_is_truncated_not_dropped`.

- **Stored technical error:** bounded raw string (≤1000 + marker), for engineers.
- **Projected API error:** the same bounded string on the run/event.
- **P2 rule (documented, not built here):** raw stored errors are **never** the default
  user-facing message — a P2 surface maps them to friendly copy. No frontend error mapping
  was built in P1.1.
