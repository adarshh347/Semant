# VISION-C · C3 — Fashion profile — report

**Date:** 2026-07-19 · **Branch:** `feat/vision-c-domain-profiles`.

## Fashionpedia benchmark → decision: local-unavailable → serverless-deferred

The roster's fashion-primary is **Fashionpedia Attribute Mask R-CNN R50-FPN**. Benchmark
on this GTX 1650 box:

| check | result |
|---|---|
| framework | needs **detectron2** — **not installed** |
| prebuilt wheel | detectron2 has **no wheel for torch 2.13.0+cu130** (very new); latest detectron2 targets much older torch |
| build from source | requires the **CUDA toolkit / `nvcc`** — **not present** on this box |
| VRAM / RAM | 4 GB VRAM · ~1 GB free RAM · 5.7 GB swap — marginal even if it built |
| checkpoint / license | Fashionpedia (Apache-2.0 data; the R50-FPN detectron2 config/weights per their repo) |

**Decision (per the build contract):** do NOT force a fragile local build and do NOT
silently substitute a different model. **Keep the exact adapter/capability identity**
(`FashionpediaAdapter`, capability `FASHION_PARSE`, resource `REMOTE`) marked
**local-unavailable** with an **explicit serverless execution path**: `infer` returns
`UNAVAILABLE` with a clear reason, and becomes available only when `FASHIONPEDIA_ENDPOINT`
is set (serverless). This preserves the contract for when a GPU endpoint is wired.

## Working garment masks — the Phase-2a segmenter, upgraded

The fashion profile's **working** garment masks come from the existing **SegFormer-clothes**
segmenter (`mattmdjaga/segformer_b2_clothes`, ATR human-parsing), honestly labeled
`detector=segformer_clothes` — the roster's own Phase-2a garment model, NOT a substitute
for Fashionpedia (which adds the 294-attribute fine parts later, serverless).

- **Upgraded to authoritative `mask_rle`** (like YOLO/architecture): each garment
  connected-component blob → `mask_rle` via `mask_geometry` + derived polygons/box +
  provenance (`adapter=segformer_clothes`, part, confidence, device).
- **Garment-part masks + parts kept SEPARATE from the coarse category** (`category` =
  garment/accessory/skin/hair; `part` = top/skirt/trousers/scarf/bag/…). Attributes stay
  empty (ATR has no attribute vocabulary — inventing them would be worse).
- Wired into `detect_regions` (C2's `_profile_chosen`): fashion profile → the garment
  pass, layered over YOLO **by geometry/precedence** (`merge_with_precedence`), never
  label-parenting. General/YOLO always runs.
- **SAM** (Refine tool) repairs boundaries — geometry, never semantic naming.
- **FashionCLIP** preserved untouched; no embeddings duplicated in C.

## Gate — multi-person fashion, garments attached by geometry, masks save/refine/recall

On the multi-person sari photo (`g_dataset_001`), the garment segmenter produced **separate
authoritative masks**: **top (0.96), scarf (0.79), trousers (0.94), bag** (garment/
accessory) plus face/arms/hair (skin/hair) — 11 regions, each `mask_rle` valid,
`adapter=segformer_clothes`, `part` separate from `category`. Each garment is its own
connected-component mask, **attached by geometry** (`merge_with_precedence` containment),
not by any shared label — asserted by
`test_garment_masks_are_geometry_not_label_parented`.

- **save/refine/recall:** these garment regions persist as `mask_rle` in
  `region_annotations` and circulate through the **identical** detect → Differential path
  browser-verified in C2 (architecture wall mask selectable in Differential) and B5 (Refine
  → confirm → reload → recall). The garment masks are the same shape of evidence on the
  same path.
- **Automated:** 4 tests (`test_vision_c3_fashion.py`) — Fashionpedia contract preserved
  (unavailable/remote, becomes available with an endpoint, explicit not-substituted);
  garment segmenter emits authoritative masks with separate parts; geometry-not-label
  parenting. Backend **105 passed**.

**Gate result: PASS** — Fashionpedia's contract is preserved with an explicit
serverless-unavailable path (no silent substitution); multi-person garment-part masks are
authoritative, geometry-attached, provenance-stamped, and save/refine/recall through the
normal route.

## Known limitations / honest gaps
- **Browser evidence sourcing gap:** the corpus has no dedicated multi-person fashion
  *post* (the sari photo is a repo fixture, not a post; the `@shrishhart` posts are pencil-
  sketch artwork). Garment masks are proven at the service level here; their circulation is
  the C2-browser-verified path. A real fashion post (eval-manifest Fashion column) gives
  dedicated screenshots — the fastest fill is the contact-sheet pass.
- Fashionpedia fine parts + attributes await a serverless GPU endpoint.
