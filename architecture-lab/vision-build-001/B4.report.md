## What was built

**Backend** — an interactive refine layer on top of B3's SAM2 adapter:
- `vision_orchestrator/refine_session.py` — a lazily-loaded singleton predictor that
  **caches the image embedding** (encode once per image, then every prompt is the warm
  path) under an `asyncio.Lock` (one heavy GPU job), running inference in a worker thread.
  Prompts arrive in normalized [0,1] coords (the stage-geometry contract) and are
  converted to pixels here. Preview never persists.
- `routers/posts.py` — three endpoints: `POST /{id}/refine-region/preview` (propose,
  no save), `.../confirm` (recompute on the cached embedding, then persist), and
  `.../unload` (free VRAM/RAM). A small per-post image-bytes cache removes the Cloudinary
  re-fetch from the warm path. **Confirm is non-destructive**: it upgrades the base
  region's identity in place (or appends), preserves curator fields, and never touches
  other regions, Grounds, Percepts or Mentions.

**Frontend** — the refinement UI:
- `components/RefineSurface.jsx` + `.css` — Point / Box / Negative tools, live preview,
  before/after, Confirm/Cancel/Clear, on the shared `viewBox = natural`,
  `preserveAspectRatio="xMidYMid meet"` contract via `useStageGeometry`. States:
  loading (spinner), proposed, empty/no-result (score < 0.2), error (incl. 503), confirmed.
  Keyboard: Enter confirm · Esc clear · P/B/N tools. `prefers-reduced-motion` disables the
  marching-ants animation.
- `pages/RefineLab.jsx` + `.css`, route `lab/refine/:postId` — a focused surface with a
  "Reload from server" control to prove persistence.

### Preventing stage pan / image drag while refining (the emphasised requirement)
By construction: every pointer handler `preventDefault`s, the stage sets pointer capture,
`.refine-img` is `pointer-events:none` + `-webkit-user-drag:none`, the stage is
`user-select:none` + `touch-action:none`, and `onDragStart` is cancelled. A point/box
gesture can never scroll or drag the image — verified live (see B4-2).

## Runtime measurements (SAM2 in the live backend, GTX 1650)

- **Cold** first prompt (model load + image encode + Cloudinary fetch): ~9.5 s.
- **Warm** prompt (image + embedding both cached): **~0.5 s** round-trip (SAM2 predict
  itself 10–170 ms; the rest is HTTP/JSON) — interactive.
- Confirm (recompute on cached embedding + persist): ~0.5–1 s.
- **Peak VRAM** with SAM2 resident in the server: ~974 MiB (model + torch context + API).
- **Unload** (`/refine-region/unload`): VRAM 974 → **162 MiB** (the ~812 MiB model
  released; 162 MiB is the persistent torch CUDA context in the long-running process, not
  the model), RAM freed to ~1.65 GiB. No destabilisation — backend stayed up, 85 tests green.
- Swap: stayed flat during the session (RAM headroom held with the single-resident +
  lazy-unload discipline).

## requirements-ml.txt CUDA reproducibility (requested check)

B0 installed torch from default PyPI, but the committed file names the cu130 index. Verified
the committed pins reproduce the CUDA build: `pip index versions` against
`https://download.pytorch.org/whl/cu130` serves **`torch 2.13.0+cu130`** and
**`torchvision 0.28.0+cu130`** (LATEST on that channel = 2.13.0+cu130), exactly the pins in
`requirements-ml.txt`. A clean `pip install -r requirements-ml.txt` therefore resolves the
CUDA wheels, not the CPU wheel. (Full-graph `--dry-run` against the PyTorch index is very
slow; the index-versions probe is the fast, decisive check.)

## Known limitations
- Cold first-prompt latency (~9.5 s) is dominated by model load + first encode; subsequent
  prompts are warm. A background pre-warm on entering refine mode would hide it.
- The refiner holds ~812 MiB VRAM while resident; the app should call `/refine-region/unload`
  on leaving refine mode (the endpoint exists; wiring the auto-unload-on-exit is a small
  follow-up).
- Automated `resize_window` did not shrink the browser viewport in this environment
  (`innerWidth` stayed 1745); the constrained viewport was verified by constraining the page
  container to 400 px, which still exercises the responsive toolbar + the stage-geometry
  ResizeObserver and coordinate registration.
