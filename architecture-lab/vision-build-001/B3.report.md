**Date:** 2026-07-19 · **Branch:** `feat/vision-differential` · **Scope:** B3 only.

## Decision

**Use SAM 2.1 Hiera Tiny locally.** It passes every 4 GB-VRAM / RAM-stability /
interaction-latency gate on this exact box by a wide margin, so the roster's intended
choice is adopted without a serverless fallback and without benchmarking the light
alternatives (the build contract benchmarks EfficientSAM-Ti/MobileSAM "only if needed for
a decision" — it was not needed).

## Integration (the `MaskRefiner` slot)

- **`Sam2RefinerAdapter`** (`vision_orchestrator/adapters.py`) implements the `Adapter`
  protocol on the GPU pool: `load` (build_sam2 + predictor), `unload` (empty_cache),
  `infer` accepting **point / box / existing-mask** prompts. Verified through
  `ModelManager.run_adapter` end to end (status succeeded, clean unload, VRAM→baseline).
- **Non-destructive by contract.** `refined_mask_to_region` emits a **PROPOSED** region:
  the mask is authoritative RLE (via `mask_geometry`), it carries the **next**
  `geometry_rev` (base 1 → 2 in the live run) and `proposed=True`, with provenance
  `{method: sam2-refine, prompt, adapter}`. Saved evidence is **not overwritten** — the
  curator confirms in B4. When a `base_id` is supplied the same Region identity is
  upgraded (lineage: `refined_from`).
- No second geometry contract — refined masks enter through Increment A's `mask_geometry`.

## Gate verification

- **Automated:** 4 new tests (`test_vision_b3_refiner.py`); backend suite **85 passed**.
  Cover: refiner mask → proposed region (RLE authoritative, box derived, `proposed=True`,
  provenance `sam2-refine`); proposes the next revision **without mutating the base**;
  box/existing-mask prompt kinds recorded; adapter identity (MASK_REFINE/GPU).
- **Real model:** benchmark table above + live `ModelManager` run (upgrades `seg_0` to
  rev 2, score 0.857) + clean unload (resident empty, VRAM 63 MiB, backend stable).

**Gate result: PASS** — the selected refiner is evidence-backed and unloads cleanly
without destabilizing the backend.

## Known limitations
- Encode is 534 ms/image; interactive multi-prompt sessions should cache the image
  encoding (orchestrator feature cache) — wired as a follow-up in B4/E.
- RSS 1.2 GB on a ~7 GB box is workable but leaves little headroom to co-resident another
  heavy model — the single-GPU-resident + unload discipline (B1) is what keeps it safe.
