# VISION-BUILD-001 · Increment B · B0 — Project CUDA migration — report

**Date:** 2026-07-19 · **Branch:** `feat/vision-differential` · **Scope:** B0 only
(project CUDA migration). Preconditions confirmed: canonical inputs present
(`VISION-BUILD-001`, `VISION-MODEL-MATRIX-001`, `VISION-ORCHESTRATOR-001` in
`architecture-lab/new files/`), Increment A commits `a1055b0` + `ebf535b` in history,
`VISION-CUDA-RUNTIME-001` probe result recorded. No application code changed in B0.

## What changed and why

The box has always had a CUDA-capable GPU; the project ran CPU-only purely because
`requirements-ml.txt` pinned the `whl/cpu` index, so `torch 2.13.0+cpu` was installed.
B0 replaces **only** Torch/Torchvision with the version-matched CUDA build the probe
verified, and hardens the requirements file so a reinstall can't regress to CPU.

## Install source & versions

- **Before:** `torch==2.13.0+cpu`, `torchvision==0.28.0+cpu` (`cuda_available=False`).
- **Command:** `pip uninstall -y torch torchvision` then
  `pip install torch==2.13.0 torchvision==0.28.0` from **default PyPI** (the linux wheel
  bundles the CUDA runtime — no system CUDA toolkit required).
- **After:** `torch==2.13.0+cu130`, `torchvision==0.28.0+cu130`; bundled **CUDA 13.0**
  runtime + **cuDNN 9.20**; +17 `nvidia-*`/`triton` runtime packages pulled in.
- **`requirements-ml.txt` hardened:** index switched `whl/cpu → whl/cu130`, and
  `torch`/`torchvision` pinned to `==2.13.0`/`==0.28.0` so `pip install -r` cannot
  silently reinstall the CPU wheel. All unrelated deps (transformers, pillow, numpy,
  ultralytics, opencv-python-headless) preserved untouched.

## B0 gate verification (project venv, not the probe venv)

Script: `scratchpad/b0_verify.py`. All assertions passed:

- `torch.cuda.is_available()` → **True**; device **NVIDIA GeForce GTX 1650**, compute
  capability **sm_75**, CUDA runtime **13.0**.
- **fp16** matmul executes on `cuda:0` (dtype float16, device cuda:0) in ~44 ms.
- **Ultralytics YOLO** (`yolo11n-seg.pt`) loads and infers on **`cuda:0`** (model params
  and `boxes.data` both on cuda:0); **3 detections** on the multi-person fixture with
  **native instance masks present**; cold **2611 ms**, warm **70 ms**.
- **Peak VRAM** during YOLO: 32 MiB allocated / 40 MiB reserved (well within 4 GB).
- **Memory release after process exit:** VRAM 63 MiB → 63 MiB (baseline) — no leak.
- **Backend suite:** `pytest backend/tests` → **52 passed** under the CUDA build.

**Gate result: PASS** — the project venv performs YOLO CUDA inference and returns
memory near baseline.

## Rollback

- Fastest: `pip install --force-reinstall -r architecture-lab/vision-build-001/B0-pre-cuda-freeze.txt`
  (the exact pre-migration environment, 80 pkgs, torch/torchvision `+cpu`).
- Or: `pip uninstall -y torch torchvision` + reinstall from `--extra-index-url
  https://download.pytorch.org/whl/cpu`, then optionally purge `nvidia-*`/`triton`.
- Post-migration snapshot for reference: `B0-post-cuda-freeze.txt`.

## Known limitations (carried into B1+)

- **4 GB VRAM** fits YOLO11n-seg / SegFormer-B2 and (per B3) is the gate SAM 2.1 Tiny
  must pass; it will **not** serve Fashionpedia R50-FPN or SAM2 base+ — those stay
  serverless per `VISION-MODEL-MATRIX-001`.
- **System RAM stays tight** (~860 MB avail, ~5.9 GB swap). CUDA Torch adds ~700 MB RSS
  on import; the orchestrator's one-heavy-GPU-job limit and unload discipline (B1) are
  load-bearing here, not optional.
