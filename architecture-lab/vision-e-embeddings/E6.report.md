# VISION-E · E6 — Resumable indexing boundary — report

**Date:** 2026-07-19 · **Branch:** `feat/vision-e-embeddings`.

## What was built (only the safe mechanics — no corpus backfill)

`backend/services/indexing_service.py` + a read-only dry-run endpoint. It reuses the machinery
already in place (deterministic `embedding_id`, `upsert` dedup, `is_stale`, ModelManager GPU
governance) and adds the batch boundary:

- **`plan_batch(posts, model, force)`** — a **dry-run**: which posts/Regions need (re)indexing
  and the estimated vectors / runtime / storage, **without fetching a single image or writing
  anything**. Freshness is judged from geometry_rev + mask + model, so planning needs no pixels.
- **`run_batch(posts, fetch_image, …, cancel, on_progress, force)`** — indexes a small explicit
  batch: **idempotent** (a post with no stale/missing Region is skipped; writes go through the
  deterministic-id upsert → no duplicate rows), **resumable** (an interrupted run re-run finishes
  only what it didn't), **cancellable** between posts, with progress events; one bad post fails
  in isolation without corrupting the batch. Embedding runs through `embed_post_regions`
  (ModelManager → single-GPU residency + `feature_key` cache + unload).
- **`GET /posts/vision/index-plan`** — the read-only dry-run over the dissected corpus. There is
  deliberately **no batch-run endpoint** — nothing here can launch a corpus-wide backfill.

Stale detection is by geometry / source / mask / model / preprocessing (a **label edit never**
triggers re-index); version isolation is by the E0 `space` — a re-preprocessed (v2) vector lives
in a different space from its v1 predecessor, so old embeddings stay query-isolated.

## Gate — interrupted resume without duplication; version isolation

`backend/tests/test_vision_e6.py` (5 tests, fake store/encoder):

- **interrupted batch resumes without duplication** — cancel after the first of three posts, then
  re-run: fresh posts are skipped, the rest finish, **every `embedding_id` is unique** (no dup
  rows), and the corpus ends fully indexed (15 vectors).
- **re-run is idempotent** — a second run over a fresh post skips it, writes nothing, and does
  **zero** whole-image encodes (no wasted GPU work).
- **geometry change re-indexes, a label edit does not** — a rename stays skipped; a new mask/rev
  is planned and re-indexed.
- **dry-run touches neither images nor store** — plan is correct and the store stays empty.
- **old vectors stay query-isolated by version** — a v1 and a v2 vector of the same Region live
  in different spaces and are each queried in isolation.

Backend suite: **203 passed** (198 + 5).

**Gate result: PASS** — an interrupted small-batch index resumes without duplication or post
corruption; fresh work is skipped; old embeddings stay query-isolated by version; and the F
backfill is sized from a real dry-run.

## Readiness / blockers for Increment F
- **Ready:** the store (E0), projections (E1), DINOv2 (E2), spaces/routing (E3), retrieval + UX
  (E5) and idempotent/resumable indexing (E6) are all in place; the backfill is ~1 min of GPU.
- **Blocker for a *useful* corpus:** most posts are **not dissected** (9/127). A corpus-wide
  embedding backfill only helps once the corpus is dissected — so F should pair dissection with
  embedding. The thin, single-domain corpus is also why OpenCLIP stays deferred (E3/E4).
